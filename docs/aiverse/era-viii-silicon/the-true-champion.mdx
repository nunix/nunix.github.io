## Rebuilding the Arena

The rigged playoff had one flaw: every tool was a prop. The `anamnesis_neurons` function returned a hardcoded list. The `system_info` function returned only the fields the General chose to expose. The models were told exactly which tool to call. It was a compliance test wearing the disguise of a capability test.

The real test required two things: open questions and real tools.

**Tool 1 — bash**: execute any shell command on the machine and return stdout. The model must decide what to run.

**Tool 2 — anamnesis_search**: query <Term>Universalis</Term> fleet memory via semantic search. The model must formulate a meaningful query.

No hints. No function names in the question. No pre-populated return values.

The questions:

- **Q1**: *"What OS and version is this machine running on?"*  
  Correct answer: SUSE Linux Enterprise Server 16.0. Only reachable via bash — `cat /etc/os-release` or equivalent.

- **Q2**: *"Who is the leader of the Warp? Check anamnesis."*  
  Correct answer: CHAOS. Only reachable via anamnesis search — the fleet memory records the Warp brain identity.

Scoring: first model to answer both correctly is champion. A faster correct model can overthrow it.

<Admonition type="info" title="⚙️ Technical Insight">
The performance gap between gemma4:12b (75 tokens, 26.5s on Q2) and gemma4-26B-A4B (1,077 tokens, 128.8s on Q2) reveals a non-obvious property of MoE models on complex reasoning chains.

With pre-defined tools and explicit prompting (the rigged playoff), MoE's active-parameter advantage holds: gemma4-26B-A4B routes the tool call decision to 4B active parameters, fires quickly, returns structured JSON. The MoE routing is cheap.

With open-ended questions and semantic search results to interpret, MoE routing fires the full processing chain for each reasoning step. The model called anamnesis twice, processed verbose fleet memory results (~1,000 characters of search output), cross-referenced entries, and synthesized a conclusion. Each step activated a different expert cluster. At 26B total parameters — even with only 4B active per token — the per-token latency accumulates across a 1,000-token reasoning chain. 1,000 tokens × higher per-token latency = 128 seconds.

gemma4:12b ran the same reasoning with 75 tokens. Fewer reasoning steps, same correct conclusion. The 12B dense model, with no MoE routing overhead, completed the chain 5× faster.

**The lesson**: MoE speed advantages are task-specific. Structured tool-call dispatch (one decision, one output) favors MoE. Open-ended multi-step reasoning over verbose retrieval results favors dense models.
</Admonition>

<Admonition type="tip" title="📚 Knowledge Transfer">

**The lesson worth keeping:** Model size does not predict benchmark performance when the benchmark is not representative of the real workload. A 7GB model beat a 27GB model on the task that actually matters — not because it is smarter, but because the task matched its architecture.

**Pattern:** Dense models outperform MoE on long reasoning chains over verbose retrieval output. MoE models outperform dense models on structured single-step tool dispatch. Know which workload you are measuring before choosing your winner.

**What we'd do differently:** Run the real test first. The rigged playoff was a pre-filter for models that could call tools at all — useful, but only if labelled honestly. "This model calls tools when instructed" is not the same as "this model is ready to be the Warp brain." One sentence of disclaimer would have saved three blog posts of correction.

**If you're building this yourself:** Write the test you actually need before you run any test at all. Ask: what will this model do in production? What inputs will it receive? What tools will it have? What constitutes a correct answer? Then build a benchmark that answers those questions — not a benchmark that produces a leaderboard.

</Admonition>
